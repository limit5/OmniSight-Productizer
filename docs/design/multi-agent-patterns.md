# OmniSight-Productizer: 多智能體架構模式 (Multi-Agent Patterns) 導入指南

本文件旨在將業界領先的五種多智能體 (Multi-Agent) 協作模式，映射並整合至我們的「嵌入式自動化開發與驗證系統」中。透過將這些模式作為架構積木，系統將能針對不同的業務瓶頸（如上下文溢出、代碼品質控管、非同步長效任務）進行精準升級，打造出高穩定度的虛擬研發團隊。

---

## 一、 生成-驗證者模式 (Generator-Verifier)
**系統定位：品質守門員與自動化模擬閉環**

* **運作原理：** 生成者負責產出結果，驗證者根據明確標準進行審核並給予回饋，直到達標為止。
* **系統對應場景：** 程式碼編譯、硬體模擬測試與 Gerrit 靜態審查。
* **實作工作流：**
  1. **[生成者] AI 開發者** 撰寫完 C/C++ 程式碼後，呼叫 `./simulate.sh`。
  2. **[驗證者] 模擬引擎** 執行編譯與測試，透過 Valgrind (記憶體檢測) 或 QEMU 產生 Log。
  3. **[驗證者] AI 審查員** 負責靜態分析，並在 Gerrit 上給予 `Code-Review -1` 附帶修改建議。
* **防呆與風險緩解：** 必須設定 **「最大重試次數 (Max Retries = 3)」**。若 AI 在解決特定指標越界或記憶體洩漏時陷入死迴圈，系統強制中斷循環，產出當前最佳版本的 `HANDOFF.md` 並標記人類主管介入。

## 二、 調度-子智能體模式 (Orchestrator-Subagent)
**系統定位：任務解析與上下文 (Context) 淨化中心**

* **運作原理：** 核心調度者負責規劃大綱，將拆解後的小任務分派給具備特定技能的子智能體，最後匯總結果。
* **系統對應場景：** 工單 (Jira/Issue) 解析與開發前置作業。
* **實作工作流：**
  1. **調度者 Agent** 讀取全新工單的規格與驗收標準 (AC)。
  2. 派發給 **Git 子智能體**：負責建立隔離的任務分支 (Branch)。
  3. 派發給 **檢索子智能體**：進入龐大的 Codebase 中尋找相關的硬體暫存器定義檔或歷史 API。
  4. 調度者將檢索結果精簡、過濾雜訊後，打包成乾淨的 Context 交給負責寫 Code 的 Agent。
* **防呆與風險緩解：** 避免調度者成為資訊瓶頸。子智能體的輸出必須經過底層工具（如 RTK 壓縮）處理，確保匯總後的資訊不會撐爆調度者的 Token 上限。

## 三、 智能體團隊模式 (Agent Teams)
**系統定位：長效非同步作戰與架構重構**

* **運作原理：** 多個智能體在共享的任務佇列中各自「搶單」，長時間、多步驟地獨立完成工作，互相不干擾。
* **系統對應場景：** 大規模底層架構重構、跨平台演算法移植 (如 x86 移植至 ARM 平台)。
* **實作工作流：**
  1. 系統啟動多個獨立的 Docker/WSL2 沙盒環境，每個沙盒內駐留一個 AI Agent。
  2. Agent A 負責重構「硬體周邊控制 (GPIO/I2C) 模組」；Agent B 負責重構「核心運算演算法模組」。
  3. 兩者在自己的環境中獨立經歷無數次的「編譯 $\rightarrow$ 模擬 $\rightarrow$ 修正」循環，持續累積該模組的專屬經驗。
  4. 最終雙方各自發起 Pull Request，由 Gerrit 進行系統級的整合測試。
* **防呆與風險緩解：** 必須在 Git Repo 層級劃定嚴格的「檔案修改權限 (Code Owners)」，避免多個 Agent 同時修改同一個核心標頭檔 (Header File) 造成災難性的 Merge Conflict。

## 四、 消息總線模式 (Message Bus)
**系統定位：事件驅動神經網與分級告警**

* **運作原理：** 透過「發布/訂閱 (Pub/Sub)」機制，智能體不直接對話，而是透過路由器 (Router) 將事件精準派發給訂閱該話題的智能體。
* **系統對應場景：** 系統 L1~L4 分級通知路由、CI/CD 流水線觸發。
* **實作工作流：**
  1. 任何系統動態皆視為「事件 (Event)」，例如：`Patch_Set_Created`、`Simulation_Failed`。
  2. **事件路由器** 接收事件並判斷嚴重等級。
  3. 若為 L2 (Warning)，將消息路由至 **IM 智能體** (推播至 Slack/Teams)。
  4. 若為 L3 (Action Required)，將消息路由至 **Jira 智能體** (自動開立阻斷工單)。
* **防呆與風險緩解：** 避免靜默崩潰。路由器本身必須具備 Dead-letter Queue (死信佇列) 機制，確保派發失敗或無人訂閱的高危險警告最終會透過 SMS 簡訊發送給值班工程師。

## 五、 共享狀態模式 (Shared State)
**系統定位：終極協作黑板與複雜除錯**

* **運作原理：** 捨棄中央調度，所有智能體共同面對一個持久化的共享黑板 (Database 或文件)，互相參考別人的線索來推進自己的工作。
* **系統對應場景：** 軟硬整合之深層系統 Crash / 綜合除錯。
* **實作工作流：**
  1. 當發生跨模組的異常時，系統建立一個針對該 Bug 的專屬 `HANDOFF.md` 或共享資料庫紀錄。
  2. **硬體 Agent** 將 Mock 測試中抓到的「異常 GPIO 電平時序」寫上黑板。
  3. **演算法 Agent** 看到黑板上的硬體時序異常，對照自身的 Log，發現是演算法的 Thread 阻塞導致硬體 Timeout，並將此發現補充上黑板。
  4. 雙方基於共享情報快速收斂問題根源。
* **防呆與風險緩解：** 設定「反應式死迴圈 (Reactive loops) 斷路器」。若黑板在 5 輪交互內皆無產生新的實質結論（例如雙方互相推卸責任），系統自動鎖定該共享狀態，並呼叫人類工程師介入仲裁。

## 總結：模式選擇建議

對於系統建置的優先順序，建議採用以下漸進式策略：
1. **初期 (Day 1)：** 優先實作 **生成-驗證者模式**。這是自動化編譯與代碼品質的基礎，沒有它，系統無法穩定產出可用代碼。
2. **中期 (Day 30)：** 導入 **調度-子智能體模式**。大幅降低單一 Agent 的認知負荷，解決大專案中的 Context Window 溢出問題。
3. **成熟期 (Day 90+)：** 逐步解鎖 **消息總線** 與 **共享狀態**，實現跨系統的動態協作與複雜問題的自主除錯能力。
