# Sandbox-Tier Audit — Per-Guild × Per-Tier Security Properties + Compliance Claims

> **BP.S.3 deliverable.** This document is the **human-readable audit doc** for
> the 4-tier sandbox model + 21-Guild admission matrix declared in
> [`backend/sandbox_tier.py`](../../backend/sandbox_tier.py) (BP.S.1) and
> narrowed at runtime by
> [`configs/sandbox_tier_policy.yaml`](../../configs/sandbox_tier_policy.yaml)
> (BP.S.2). It is intended to be cited **verbatim** by Phase D auxiliary
> compliance modules (`backend/compliance_matrix/{medical,automotive,industrial,military}.py`).
>
> **Drift guard.** BP.S.6 will add a parameterised test
> (`backend/tests/test_sandbox_tier_policy.py::test_audit_doc_matches_runtime_matrix`)
> that parses the per-Guild table in §3 below and compares it cell-by-cell
> against `GUILD_TIER_ADMISSION_MATRIX`. If this doc and the code disagree,
> CI fails red — neither side gets to "win" silently.

## 0. ⚠️ Auxiliary-tool disclaimer (Phase D mandatory header)

This document, and any Phase D auxiliary-check module that cites it, is
**not** a regulatory certification artefact. Any compliance claim made
below is:

* an **engineering-level mapping** between OmniSight's sandbox boundary
  and the named standard's clauses;
* meant to assist a **human certified engineer / certified auditor /
  third-party legal reviewer** in their own assessment;
* **NEVER** a substitute for that human assessment.

API responses generated from any module that cites this doc must carry
`audit_type="advisory"` and `requires_human_signoff=true` — see
[blueprint-v2-implementation-plan.md §Phase D](blueprint-v2-implementation-plan.md)
clauses for the contractual schema.

> **R12 — gVisor is nominal, not actual** (BP.S.5 risk register entry).
> The Tier 1 / Tier 2 sandboxes described below currently run on **plain
> Docker** in production, not gVisor. The gVisor reference in
> `backend/sandbox_capacity.py` is a **DRF cost-weight constant**, not a
> runtime selector. **No compliance claim in this document may be read as
> "gVisor user-space kernel isolation is provided"** until BP.W3.13
> (Phase U) lands the actual gVisor adoption. Standards that require
> kernel-level isolation (e.g. IEC 62304 Class C, ISO 26262 ASIL D) MUST
> rely on additional out-of-band controls; the Tier model alone is
> insufficient evidence.

## 1. The 4-tier model — security properties (1-row summary)

Verbatim mirror of [`tiered-sandbox-architecture.md`](tiered-sandbox-architecture.md)
§I and [`backend/sandbox_tier.py::SandboxTier`](../../backend/sandbox_tier.py).

| Tier | Name | Network | Hardware | Persistence | Threat model addressed |
|------|------|---------|----------|-------------|------------------------|
| **T0** | Control Plane (cloud, **no sandbox**) | Full egress + ingress to internal LAN | None | Long-lived (orchestrator state, audit chain) | (a) holds API keys / Gerrit SSH keys; (b) **MUST NOT execute AI-generated payloads** — protective property is "AI-generated bytes never reach a code-executing context inside T0". |
| **T1** | Strict Sandbox (ephemeral microVM-class container) | **Air-gapped** except whitelisted Git server | None | Ephemeral (destroyed after each task) | (a) Cgroups cap 4 vCPU / 8 GB → prevents OOM / runaway loops crashing host; (b) reproducibility (no internet during compile = no transitive supply-chain pull); (c) any Segfault / panic / undefined-behaviour stays inside the disposable container. |
| **T2** | Networked Sandbox (Docker w/ egress) | Egress to public Internet allowed; **VPC-isolated from internal LAN** | None | Ephemeral | Mitigates prompt-injection → internal-network-scan attack: AI may pull npm / pip / threat-feed data, but a compromised AI cannot pivot to internal databases. |
| **T3** | Hardware Bridge (bare-metal host wired to EVK) | RPC-only (Hardware Daemon API) | Real EVK + USB / UART / I2C / SPI | Long-lived (the lab box) | Agent **NEVER** gets a shell on this host: only structured JSON RPC like `{"action":"flash_board","firmware_url":"…"}`. The Daemon validates parameters and refuses dangerous primitives (`rm`, `dd` on system partitions, etc.). |

**Cross-reference — capacity dimension is orthogonal.** This document
covers the *admission* axis (who is allowed to run where). The *cost*
axis (how much of each Tier each Guild can consume) lives in
[`backend/sandbox_capacity.py`](../../backend/sandbox_capacity.py); they
share Tier names but answer different questions.

## 3. Per-Guild × Tier admission table — security properties + compliance claims

> **Source-of-truth.** This table is structurally derived from
> `GUILD_TIER_ADMISSION_MATRIX` in [`backend/sandbox_tier.py`](../../backend/sandbox_tier.py).
> Operator narrowing (`configs/sandbox_tier_policy.yaml`, BP.S.2) may
> *remove* admitted tiers per deployment but never *add*. The drift-guard
> test in BP.S.6 fails CI if any row's "Admitted Tiers" cell does not
> equal the runtime matrix.
>
> **Reading the rows.** "Admitted Tiers" lists every Tier the Guild
> structurally *may* run in. Each subsequent column documents (a) the
> security property the (Guild, Tier) pair achieves, and (b) which
> auxiliary compliance claim the Phase D modules may surface based on
> that property. **All claims are advisory** — see §0 disclaimer.

### 3.1 Cloud-brain Guilds (T0-only or T0+T2)

touch hardware, never execute AI-generated payloads.

|-------|---------------|------------------------------|--------------------------------|
| `architect` | **T0, T2** | T0: control-plane reasoning over architecture diagrams + ADRs; no payload execution. T2: outbound to fetch reference architectures / RFCs / SOTA papers; VPC-isolated from internal LAN. | IEC 62304 §5.1 (planning) — design-phase artefacts produced inside T0/T2 boundary. | `sa_sd` | **T0, T2** | Same as `architect`. T0 hosts SA/SD analysis tooling (UML, sequence diagrams). ## 4. The 84 (Guild × Tier) cells — denial side

The 21 × 4 = 84 conceptual (Guild, Tier) cells minus the 53 admitted
cells above = **31 cells where dispatch is denied** by the structural
matrix. Each denial is a **policy bug indicator**, not just a routing
failure: PEP Gateway (BP.S.4) raises `GuildTierViolation` with the
offending pair + permitted set so the audit log is self-contained.

The denied cells are not enumerated row-by-row here — they are the
structural complement of §3. The drift-guard test in BP.S.6 covers both
sides: every admitted cell present in the doc table, every denied cell
absent, byte-equal to the runtime matrix.

**Notable denials worth surfacing for compliance reviewers:**

* `auditor` × {T1, T2, T3}: KS.1 audit-chain integrity. Compliance
  significance: ISO 27001 A.12.4 + NIST AU-9 (protection of audit
  information) — the auditor cannot become the attacker.
* `gateway` × {T1, T2, T3}: secrets live in T0; gateway never executes
  AI-generated payloads. Compliance significance: NIST AC-3 / SC-12 +
  ISO 27001 A.10 (key management) — secret-touching code path is
  isolated from payload-executing code path.
* `red_team` × {T0, T3}: RedTeam never gets control-plane execution
  (T0) or real-hardware access (T3). Compliance significance: NIST
  CA-8 + ISO 27001 A.14.2.8 — pentest scope is structurally bounded.
* `pm` / `ux` × {T1, T2, T3}: planning Guilds have no execution
  environment. Compliance significance: HIPAA §164.310(d) +
  ISO 13485 §7.3 — strategic / design data does not touch
  payload-executing tiers.
* All cloud-brain Guilds × {T1, T3}: brain-vs-hands separation is the
  core architecture invariant. Compliance significance: ISO 26262
  Part 2 §6.4 + IEC 62304 §5 (organisational separation — a Guild
  responsible for *deciding what to build* is structurally distinct
  from one responsible for *building it*).

## 5. Operator-narrowed deployments — additional claims

When `configs/sandbox_tier_policy.yaml` (BP.S.2) is configured to
*narrow* a Guild's admission (e.g. air-gapped deployment removes T2
from every Guild), additional compliance claims become available.
Phase D modules that consume this doc MAY emit the additional claims
below **only if** the operator narrowing is verified at runtime by the
BP.S.4 loader (which BP.S.6 tests will check).

| Operator narrowing | Additional advisory claim |
|--------------------|---------------------------|
| Every Guild stripped of T2 (air-gap mode) | NIST SP 800-53 SC-7 (boundary protection) — **outbound network exfiltration path is structurally absent**. Suitable for classified-data deployments referencing MIL-STD-882E §4.4. |
| `red_team` `allowed_tiers: []` (incident-response lockdown) | ISO 27001 A.16.1.5 (response to information security incidents) — adversarial Guild quiesced for the duration of the incident; auditable via the `reason:` field in the yaml. |
| `bsp` / `hal` / `algo_cv` / `isp` / `audio` stripped of T3 (lab maintenance) | DO-178C §11.4 (problem report) — hardware-affecting dispatch quiesced; affected work items roll forward as `[blocked-on-hardware]` in the queue, no silent timeout. |
| `intel` `allowed_tiers: []` (air-gapped, no threat feed) | NIST SP 800-53 RA-5 control marked as **compensating control required** — operator must point to an offline CVE-import process; without it, vulnerability monitoring is degraded. |

## 6. What this doc does NOT claim

Concise list of common misreadings to head off during legal review:

1. **NOT** "OmniSight is IEC 62304 / ISO 26262 / DO-178C / MIL-STD-882E
   certified." It is not. This doc maps OmniSight's sandbox boundary to
   clauses of those standards as engineering evidence; certification is
   a separate human-led process.
2. **NOT** "gVisor user-space-kernel isolation is in place." It is
   nominal until BP.W3.13 (Phase U). See §0 R12 disclaimer.
3. **NOT** "AI-generated code is verified safe by being sandboxed."
   Sandboxing bounds the *blast radius* of unsafe code; correctness of
   the code itself is verified by the QA Guild + human review per the
   Phase D module's `requires_human_signoff=true` contract.
4. **NOT** "Tier admission alone discharges the standard's clause."
   It is one piece of evidence; the named human certified engineer /
   auditor remains the load-bearing signature.
5. **NOT** "the operator narrowing in §5 is sufficient on its own."
   It is necessary for some deployments (air-gap, lockdown) but always
   composes with other controls (RBAC, encryption-at-rest, audit chain
   integrity, etc.) — see KS.* + BP.B.* epics for the full control set.
