# MFA Enrollment Guide

**Audience**: OmniSight operators and admin users.
**Self-service**: complete enrollment yourself in ~5 minutes.
**Status**: Optional today. Will become mandatory under Phase D / SOC 2 / ISO 27001 (CL Priority series).

---

## What MFA is in OmniSight (and what it isn't)

OmniSight implements **K5 — MFA** with three methods:

| Method | What it is | Use case |
|---|---|---|
| **TOTP** | 6-digit code rotates every 30 seconds, generated by an authenticator app on your phone | Primary; works offline; cheap |
| **WebAuthn (passkey)** | Hardware key (YubiKey) or biometric (Touch ID / Face ID / Windows Hello) | Strongest; phishing-resistant; no typing |
| **Backup codes** | 8-10 single-use codes printed once at TOTP enrollment | Emergency access if you lose your phone |

**Not implemented (by design)**:
- ❌ Email-based 6-digit code — phishable, depends on email account integrity
- ❌ SMS code — NIST 800-63 deprecated; vulnerable to SIM-swap attacks
- ❌ Push notifications — adds vendor dependency without security gain

The code in `backend/mfa.py` is built on `pyotp` (TOTP) + `py_webauthn` (WebAuthn). All three method types share one challenge endpoint at `/auth/login` step 2.

## Enroll TOTP (recommended first method, ~3 min)

### Prerequisites

Pick one authenticator app on your phone (free):

| App | Best for |
|---|---|
| **1Password / Bitwarden / Dashlane** | If you already use a password manager — TOTP integrates with passwords, syncs across devices automatically |
| **Authy** | Cross-device cloud sync (rebuild phone without losing codes) |
| **Google Authenticator** | Simple, syncs via Google account |
| **Microsoft Authenticator** | If you use Microsoft 365 / Azure AD already |

**Don't**: hand-copy the secret string to a paper note — defeats the device-bound security model.

### Enrollment steps

1. Log in to OmniSight via username + password as usual.
2. Navigate to **Account Settings** → **Security** (component path: `components/omnisight/mfa-management-panel.tsx`).
3. Click **Add TOTP Method**. The UI displays:
   - A QR code
   - A secret string (base32, e.g. `JBSWY3DPEHPK3PXP...`)
4. Open your authenticator app on the phone:
   - **1Password / Bitwarden**: open the password entry for OmniSight → **Edit** → **Add One-time Password** → **Scan QR code**
   - **Authy / Google Authenticator / Microsoft Authenticator**: tap **+ → Scan QR code**
5. Point the phone camera at the QR code. The app immediately starts generating 6-digit codes that rotate every 30 seconds.
6. Back in the OmniSight UI, type the **current 6-digit code** into the verification box. Click **Verify**.
7. On success, the UI displays **8-10 backup codes**. **Save them now** — see "Backup codes handling" below.
8. Click **Done**. TOTP is enrolled.

### Verify enrollment worked

Log out of OmniSight. Log in again with email + password. After password verifies, you should be redirected to `/mfa-challenge` asking for your 6-digit code. Open authenticator app → enter current code → submit. You're in.

If you're NOT redirected to MFA challenge after enrollment, something is wrong:
- Check `backend/mfa.py::has_verified_mfa(user_id)` returns True for your account
- Check `OMNISIGHT_REQUIRE_MFA` env flag is not set to "false" with override

## Enroll WebAuthn / passkey (optional second method, ~2 min)

### What you need

One of:
- **Hardware security key** (YubiKey 5 series, Google Titan, SoloKeys) — USB-A or USB-C, ~$30-60 each
- **Platform authenticator** (Touch ID on Mac, Face ID / Touch ID on iPhone, Windows Hello, Android fingerprint) — built-in, $0
- **Phone passkey** (iOS 16+ / Android with Google Password Manager) — built-in

### Enrollment steps

1. Same path: **Account Settings** → **Security** → **Add WebAuthn Method**.
2. Browser prompts you to authenticate with your chosen method:
   - **Hardware key**: insert into USB port + tap (button or capacitive touch)
   - **Touch ID / Face ID**: place finger on sensor or look at camera
   - **Windows Hello**: PIN or biometric prompt
   - **Phone passkey**: scan QR code with phone, authenticate on phone
3. Browser sends signed challenge response to OmniSight backend.
4. UI confirms enrollment. Done — no code typing involved.

### Why also enroll WebAuthn if you already have TOTP

- **Phishing-resistant**: WebAuthn binds to origin (sora.services), can't be tricked by fake login pages. TOTP can be phished if attacker controls a fake site.
- **Faster login**: tap key once vs. open app + read code + type code. ~5x faster.
- **Backup**: if your phone (with TOTP app) dies, you still have hardware key.

You can enroll **multiple WebAuthn methods**: e.g., one YubiKey on your desk + Touch ID on Mac + iPhone passkey. Lose one, still have others.

## Backup codes handling (CRITICAL — read this carefully)

When TOTP is enrolled, OmniSight generates 8-10 single-use codes like `J9X4-K2P7-MNQ8-...`. They are your **emergency access** if:

- You lose your phone
- TOTP app data is wiped (factory reset)
- Authenticator app is uninstalled

### Where to store backup codes

Pick **two** of these (redundant storage):

| Storage | Pros | Cons |
|---|---|---|
| **Password manager (1Password / Bitwarden / Dashlane) "secure note"** | Encrypted at rest, synced across devices | Single point of failure if password manager itself locks you out |
| **Printed paper, locked drawer / safe** | Survives all-digital-disasters | Physical; can be lost / damaged; harder to read after years |
| **Encrypted USB drive in a safe place** | Strong encryption | Drive can fail; finding it after 2 years is the test |
| **Bank safe deposit box (paper)** | Disaster-tier durability | Slow access (business hours, branch trip) |

**Don't**:
- ❌ Save in OmniSight workspace (defeats the purpose if OmniSight is down)
- ❌ Save in plain text on the same machine you log in from (no entropy advantage if attacker has machine)
- ❌ Email the codes to yourself (in plain — phishable email = phishable codes)
- ❌ Screenshot to phone gallery (phone gallery often syncs to cloud unencrypted)

### Use backup codes when

- Phone is lost / broken / wiped → use one backup code to log in → IMMEDIATELY enroll a new TOTP method while you're in.
- Each backup code is **single-use**. After using one you have N-1 left. After using ~half, regenerate a new set (`Account Settings → Security → Regenerate Backup Codes` — invalidates all previous codes).

## Recovery scenarios

### Scenario 1: Lost phone, have backup codes

1. Log in with email + password
2. At MFA challenge, click "Use backup code"
3. Enter one unused backup code → log in
4. Go to Security settings → **Remove TOTP** (the now-orphaned one)
5. Get new phone → enroll fresh TOTP
6. Regenerate backup codes (old ones still work but it's hygiene)

### Scenario 2: Lost phone AND lost backup codes

1. **You can't self-recover.** Email operator (sora) for emergency unlock.
2. Operator path: log into Atlassian admin / GitLab / etc. as sora → go to OmniSight admin → reset MFA for affected user
3. Affected user gets a one-time bypass URL valid 30 minutes
4. They re-enroll TOTP + new backup codes

### Scenario 3: Lost phone, only have WebAuthn key (no TOTP)

1. Use WebAuthn key on the login page
2. Go to Security settings → **Remove TOTP** (if you had one) or **Add new TOTP**
3. WebAuthn-only is fully supported; you don't need TOTP if you have WebAuthn enrolled.

### Scenario 4: Re-enroll after key change (compromise / rotation)

If you suspect your authenticator app data is compromised (e.g. phone was unlocked + scanned by an attacker):
1. Log in with the suspicious method (one last time)
2. Go to Security → **Remove TOTP**
3. Add new TOTP with a fresh QR code (different secret)
4. **Regenerate backup codes** — old ones are voided
5. **Change password** too — Q.1 will auto-rotate all peer sessions

## Operator-specific notes

### sora (primary admin)

- You are the **operator** for OmniSight infrastructure
- You should enroll **TOTP + at least 1 WebAuthn method** (redundancy)
- You hold the emergency MFA reset capability for other users — write a runbook entry for this in `docs/runbook/` (Phase 5 prep)
- If you lose all your MFA + backup codes, recovery is via Atlassian admin → reset → bootstrap admin re-enrollment (operator's-operator scenario)

### Future agents (claude-bot / codex-bot)

Bot accounts **do not need MFA** because they authenticate via PAT (JIRA) / SSH key (Gerrit) / Token (GitLab) — not via the OmniSight login flow. PAT is itself a strong authentication factor, so adding TOTP would be redundant.

Phase 3+ when bots interact with OmniSight via the same login flow as humans, this changes — but that's specific Phase 3 work, not current.

### Other future humans (lint-bot reviewer / merger-agent / etc.)

Each human gets own MFA setup. The `non-ai-reviewer` group (per ADR 0003) requires all members enrolled before they can cast +2 votes, post-Phase 3 submit-rule.

## Commercial Launch / Compliance implications

### When MFA becomes mandatory

| Trigger | When | Effect |
|---|---|---|
| `OMNISIGHT_REQUIRE_MFA=true` env knob set | Operator decision | Login flow forces enrollment for users who haven't enrolled |
| SOC 2 Type I audit prep (CL.1) | ~3 months before audit | Auditor mandates this as a control |
| ISO 27001 certification (CL.2) | Same window as SOC 2 | Same control requirement |
| First medical / automotive / military customer (Phase D) | Customer security review | Enterprise contracts typically require MFA in your security questionnaire |

### When the env knob is enforced

The flow is:
1. User submits email + password
2. Backend checks `has_verified_mfa(user_id)`:
   - If True → proceed to MFA challenge as today
   - If False AND `OMNISIGHT_REQUIRE_MFA=true` → block login + redirect to `/account-locked` with message "MFA enrollment required" and a one-time enrollment URL
3. User completes enrollment (forced TOTP setup) → can log in

This means the **breaking change for users without MFA** is a one-time forced enrollment, not a permanent lockout. Once MFA is set up, normal login resumes.

### Implementation references

- `backend/mfa.py` — TOTP + WebAuthn + backup codes core
- `backend/routers/mfa.py` — REST endpoints (`/mfa/enroll`, `/mfa/verify`, `/mfa/regenerate-backup`)
- `backend/routers/auth.py:425-462` — login flow integration with MFA challenge
- `app/mfa-challenge/page.tsx` — challenge page UI
- `components/omnisight/mfa-management-panel.tsx` — settings UI for enrollment / remove
- `components/omnisight/mfa-method-tabs.tsx` — method picker (TOTP / WebAuthn / backup)

- `OMNISIGHT_REQUIRE_MFA` env var — global enforcement flag (default unset)
- `OMNISIGHT_WEBAUTHN_RP_NAME` env var — WebAuthn relying party display name

## Quick reference — TL;DR

1. Pick authenticator app (1Password / Authy / Google Authenticator).
2. Account Settings → Security → Add TOTP → scan QR → enter 6-digit code.
3. Save the 8-10 backup codes in a password manager + paper backup.
4. (Recommended) Add WebAuthn passkey (Touch ID / YubiKey) for faster + phishing-resistant login.
5. Test: log out + log back in, verify MFA prompt appears.
6. Done. Login flow now: email + password → 6-digit code (or WebAuthn touch) → in.
